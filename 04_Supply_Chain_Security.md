# What is SBOM and Why It's Important
- SBOM means Software Bill of Materials.
- It is like an ingredients list for software. It tells you what components exist inside an application or container image, such as libraries, packages, dependencies, versions, and relationships between components
- Without SBOM, you may run software without knowing what is inside it.

Example: 
```text
Application: my-api:v1

Contains:
- node:20-alpine
- express 4.18.2
- lodash 4.17.21
- openssl 3.x
- libc packages
```
With SBOM, you can answer:
```text
What packages are inside this image?
Which versions are used?
Do any packages have known CVEs?
Which app is affected by a new vulnerability?
Are we using risky or outdated dependencies?
```

# Minimize Base Image Footprint
- **Minimize base image footprint** means: make your container images smaller, cleaner, and safer before deploying them to Kubernetes.
- The smaller the image, the fewer packages it contains.
- Fewer packages means:
	- less attack surface
	- fewer CVEs
	- faster image pull
	- faster deployment
	- easier scanning
	- less unnecessary software inside the container

- below gives an example where scanning `httpd` with Trivy shows many vulnerabilities, while `httpd:alpine` shows fewer or no vulnerabilities.
- Use the smallest image that can safely run your application.
- Good order from smallest to largest:
	- scratch
	- distroless
	- alpine
	- slim
	- full ubuntu/debian

- Minimizing base image footprint means reducing unnecessary packages and tools inside your container image, so the Kubernetes workload has a smaller attack surface and fewer vulnerabilities.

```sh
trivy image httpd
```

# SBOM Format
- A Software Bill of Materials (SBOM) is a comprehensive inventory of all components, libraries, and modules within a piece of software. It is crucial for ensuring software supply chain security by providing transparency and allowing organizations to identify and address vulnerabilities efficiently. 
- There are two widely adopted Software Bill of Materials (SBOM) formats: `SPDX` and `CycloneDX`. Both formats have distinct focuses—`SPDX` emphasizes licensing and legal compliance, while `CycloneDX` concentrates on security aspects such as identifying vulnerabilities and managing supply chain risks.
- You use SPDX / CycloneDX SBOM because you need to know exactly what is inside your image or application.
- To find vulnerable packages: Imagine a new critical CVE appears in openssl,
	- Without SBOM: You must manually check every image.
	- With SBOM: Search the SBOM files and quickly know which images contain openssl.
- SBOMs enable:
	- Better management of software dependencies,
	- Improved compliance with licensing requirements,
	- Quick responses to security threats.
- By adopting SBOMs, organizations can enhance their security posture and reduce risks associated with third-party software components.

```text
Build image
↓
Generate SBOM
↓
Scan image / SBOM
↓
Find CVEs
↓
Fix image
↓
Deploy safer workload to Kubernetes
```

## SPDX Format
- SPDX is a comprehensive SBOM format that organizes information into several key sections to ensure thorough documentation of software packages
- SPDX is an open standard for SBOM information like provenance, license, security, and supply-chain metadata:
	1. **Document Information**: Contains metadata about the SPDX document, including creator details, creation date, and version.
	2. **Relationships**: Defines how various components of the SBOM interrelate (e.g., file-to-package or dependency relationships).
	3. **Package Information**: Provides specific software package details such as name, version, supplier, and verification information (e.g., checksums).
	4. **Snippets**: Captures smaller sections of code or components, including excerpts from open-source libraries.
	5. **File Information**: Tracks individual files within the package.
	6. **Additional Metadata**: Offers extra details like notes, licensing information, and review records to ensure the SBOM’s accuracy.

**Example: SPDX Document for NGINX Package**
Below is an example JSON snippet representing an SPDX document for the NGINX package. This document follows the SPDX 2.3 specification and was generated by ANKOR using the SIFT tool.
```json
{
  "spdxVersion": "SPDX-2.3",
  "dataLicense": "CC0-1.0",
  "SPDXID": "SPDXRef-DOCUMENT",
  "name": "nginx",
  "documentNamespace": "https://anchore.com/syft/image/nginx-2a35db70-da10-45cd-b82d-00921857780f",
  "creationInfo": {
    "licenseListVersion": "3.25",
    "creators": [
      "Organization: Anchore, Inc",
      "Tool: syft-1.13.0"
    ]
  },
  "created": "2024-09-24T18:17:42Z"
}
```

**How to use SPDX**
Example with Trivy:
```sh
trivy image --format spdx --output pihole-sbom.spdx pihole/pihole:latest
# or 
trivy image --format spdx-json --output pihole-sbom.json pihole/pihole:latest
# Read the SPDX file:
cat pihole-sbom.json | jq .
# Search for packages: 
cat pihole-sbom.json | grep -i openssl
```
SPDX output usually helps answer:
- What packages exist?
- Which versions?
- What licenses?
- Who created the SBOM?
- What is the package relationship?

## CycloneDX Format 
- CycloneDX is designed as a lightweight SBOM format that places special emphasis on security and compliance. It is particularly useful for identifying vulnerabilities and managing component dependencies
- CycloneDX is an OWASP Bill of Materials standard focused on supply-chain risk reduction and supports use cases like SBOM, VEX, CBOM, SaaSBOM
- Key sections in the CycloneDX format include:
	1. **BOM Metadata**: Contains general information about the SBOM such as version, timestamp, and creator details.
	2. **Components List**: Enumerates the individual parts or modules within the software.
	3. **Vulnerabilities**: Lists known security vulnerabilities associated with the components.
	4. **Software Services**: Details any services associated with the software.
	5. **Annotations**: Provides additional notes or contextual information.
	6. **Dependencies and Extensions**: Offers further insights into component dependencies and extended metadata.

**Example: CycloneDX BOM in JSON**
The following JSON snippet illustrates a CycloneDX BOM, showcasing schema information, metadata, and component details.
```json
{
  "$schema": "http://cyclonedx.org/schema/bom-1.6.schema.json",
  "bomFormat": "CycloneDX",
  "specVersion": "1.4",
  "serialNumber": "urn:uuid:e7f6caab-6589-430d-bb7f-0076d23e9efb",
  "version": 1,
  "metadata": {
    "timestamp": "2024-09-24T18:46:28Z",
    "tools": {
      "components": [
        {
          "type": "application",
          "author": "anchore",
          "name": "syft",
          "version": "1.13.0"
        }
      ]
    }
  },
  "component": {
    "bom-ref": "eb2d7db1213e6155",
    "type": "container",
    "name": "nginx",
    "version": "sha256:edf555d07d2ddeb6b616d9024442feac12a91310c9a156fa6f60cd602881a"
  },
  "properties": [
    {
      "name": "syft:image:labels:maintainer",
      "value": "NGINX Docker Maintainers <docker-maint@nginx.com>"
    }
  ],
  "components": [
    {
      "bom-ref": "pkg:deb/debian/adduser@3.134?arch=all&distro=debian-12&package-id=8a498975e59f569c2",
      "type": "library",
      "publisher": "Debian Adduser Developers <adduser@packages.debian.org>"
    }
  ]
}
```

**How to use CycloneDX**
```sh
# Generate CycloneDX SBOM:
trivy image --format cyclonedx --output pihole-sbom.json pihole/pihole:latest
# Read it:
cat pihole-sbom.json | jq . 
# Search for a package:
cat pihole-sbom.json | jq '.components[]?.name' | grep libssl3
```

## Comparison Between SPDX and CycloneDX

Both SPDX and CycloneDX are powerful SBOM formats, each with their own set of strengths. The table below summarizes their key differences:

| Feature             | SPDX                                                                                                                                                                                                                                                                                                  | CycloneDX                                                 |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| Format & Complexity | Extensive metadata with focus on licensing and compliance. Available in JSON and RDF formats - licenses, compliance, package metadata, audit Lightweight format focusing on security and vulnerabilities. Available in JSON and XML formats - security, vulnerability management, supply chain risk - |                                                           |
| Security Focus      | Detailed license data and compliance metrics.                                                                                                                                                                                                                                                         | Emphasizes vulnerability tracking and dependency manageme |
| Ease of Use         | More complex due to extensive metadata coverage.                                                                                                                                                                                                                                                      | Simpler and more focused on security and complia          |

## SBOM Workflow​ 
```text
Source Code
   ↓
Build Application
   ↓
Build Container Image
   ↓
Generate SBOM
   ↓
Scan SBOM / Image
   ↓
Store SBOM
   ↓
Deploy to Kubernetes
   ↓
Monitor for new CVEs
```

```sh
# 1. Generating an SBOM:
# Generate an SBOM for the docker.io/kodekloud/webapp-color:latest image in SPDX format using Syft and store it in /root/webapp-spdx.sbom
syft docker.io/kodekloud/webapp-color:latest -o spdx > /root/webapp-spdx.sbom

# To generate an SBOM for the docker.io/kodekloud/webapp-color:latest image in CycloneDX JSON format, run the command below:
syft docker.io/kodekloud/webapp-color:latest -o cyclonedx-json > /root/webapp-sbom.json

# After storing the SBOM securely, the next step is vulnerability scanning. Grype is an excellent tool for this purpose. Follow these steps:
# 2. Scanning the SBOM
# Analyze the /root/webapp-sbom.json SBOM using Grype to generate a vulnerability report and store it in /root/grype-report.json.
grype sbom:/root/webapp-sbom.json -o json > /root/grype-report.json

~ ➜  grype sbom:/root/webapp-sbom.json -o json > /root/grype-report.json

   ├── by severity: 21 critical, 91 high, 83 medium, 11 low, 1 negligible  # the total number of Critical vulnerabilities is 21 
   └── by status:   122 fixed, 85 not-fixed, 0 ignored

cat grype-report.json | jq -e '.matches[] | select(.vulnerability.id == "CVE-2022-48174")' | grep -i severity
```

# KubeLinter
- KubeLinter is a static analysis tool that scans Kubernetes YAML manifests and Helm charts before deployment. It detects security problems, production-readiness issues, and configuration mistakes.

Given a Deployment like:
```yaml
spec:
  replicas: 1
  template:
    spec:
      containers:
        - name: app
          image: my-app:latest
```
KubeLinter may report:
- Only one replica, meaning no redundancy
- Use of the latest image tag
- Missing CPU/memory requests and limits
- Missing liveness and readiness probes
- Missing or insecure securityContext
- Containers allowed to run as root
- Excessive container privileges
- Sensitive data placed outside Kubernetes Secrets

Run it against one manifest:
```sh
kube-linter lint deployment.yaml
# Scan every manifest in a directory:
kube-linter lint ./k8s
# Scan Helm charts:
kube-linter lint ./charts/my-app
```
CI/CD use:
```text
Commit manifests
      ↓
Run KubeLinter
      ↓
Checks pass?
   Yes → build/deploy
   No  → stop pipeline and fix YAML
```

| Tool                     | Purpose                                                     |
| ------------------------ | ----------------------------------------------------------- |
| **KubeLinter**           | Scans Kubernetes YAML and Helm configuration                |
| **Trivy**                | Scans container images and dependencies for vulnerabilities |
| **Admission controller** | Enforces policies when resources enter the cluster          |
| **Falco**                | Detects suspicious behavior at runtime                      |

# Image Security
## Understanding image names
- A complete image reference follows this structure: `registry/account-or-project/image:tag`
- Example: `docker.io/library/nginx:1.27`
- When you specify only: `image: nginx`, Kubernetes effectively interprets it as: `image: docker.io/library/nginx:latest`
	- `docker.io`: registry
	- `library`: Docker Hub’s official-image namespace
	- `nginx`: repository
	- `latest`: default tag when none is specified

## Public versus private registries
| Registry | Example                           | Purpose                              |
| -------- | --------------------------------- | ------------------------------------ |
| Public   | `docker.io/library/nginx`         | Publicly accessible images           |
| Private  | `private-registry.io/apps/api:v1` | Internal and restricted applications |
Private registries protect proprietary images and control who can push or pull them. Examples include:
- AWS ECR
- Azure Container Registry
- Google Artifact Registry
- Harbor
- Private Docker registries

### Authenticate from Docker
To access a private registry manually:
```sh
docker login private-registry.io
docker pull private-registry.io/apps/internal-app:v1
```
However, a Kubernetes worker pulling the image also needs registry credentials.
### Create an image-pull Secret
Create a special Kubernetes Secret of type `kubernetes.io/dockerconfigjson`:
```sh
kubectl create secret docker-registry regcred \  # regcred: just a secret name 
  --docker-server=private-registry.io \
  --docker-username=registry-user \
  --docker-password=registry-password \
  --docker-email=registry-user@example.com
 
# Verify it: 
kubectl get secret regcred
kubectl describe secret regcred
```
The Secret must normally exist in the same namespace as the Pod.

### Reference it from the Pod
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: internal-app
spec:
  containers:
    - name: app
      image: private-registry.io/apps/internal-app:v1
  imagePullSecrets: 
    - name: regcred # Use the credentials stored in the Secret named regcred when pulling this Pod’s container images. 
```

# Whitelist Allowed Registries - Image Policy Webhook
How to ensure Kubernetes accepts images only from approved registries: 
How enforcement works:
```text
kubectl creates Pod
→ Authentication
→ Authorization
→ Admission controller
→ Webhook checks image registry
→ Allow or reject Pod
```
Example policy:
```text
internal-registry.io/myapp:v1 → Allowed
docker.io/random/myapp:v1    → Rejected
```
**Available approaches**
1. Custom ValidatingAdmissionWebhook
2. [[#^opaimage|^OPA Gatekeeper with Rego policies]]
3. [[#^ipw|^Built-in ImagePolicyWebhook admission controller]]

**OPA/Rego** ^opaimage
```text
package kubernetes.admission

deny[msg] {
  input.request.kind.kind == "Pod"
  image := input.request.object.spec.containers[_].image
  not startswith(image, "internal-registry.io/")
  msg := sprintf("Image '%s' is not from a trusted registry", [image])
}
```
This checks every normal container and rejects images not beginning with the approved registry address.

**ImagePolicyWebhook** ^ipw

`ImagePolicyWebhook` is a built-in **validating admission controller**. It asks an external HTTPS service whether the images should be accepted.

1. Admission configuration `admission-config.yaml`: 
	```yaml
	apiVersion: apiserver.config.k8s.io/v1
	kind: AdmissionConfiguration
	plugins:
	  - name: ImagePolicyWebhook
	    configuration:
	      imagePolicy:
	        kubeConfigFile: /etc/kubernetes/image-policy-kubeconfig.yaml
			 # This tells the ImagePolicyWebhook admission controller: Use this kubeconfig file to learn how to connect and authenticate to the external image-policy webhook server
	        allowTTL: 50
	        denyTTL: 50
	        retryBackoff: 500
	        defaultAllow: false
	```

	
	| Setting          | Meaning                                       |
	| ---------------- | --------------------------------------------- |
	| `kubeConfigFile` | Connection and TLS details for the webhook    |
	| `allowTTL`       | Seconds to cache allowed decisions            |
	| `denyTTL`        | Seconds to cache denied decisions             |
	| `retryBackoff`   | Milliseconds between retries                  |
	| `defaultAllow`   | What happens if the webhook cannot be reached |

2. KubeConfig file referenced above might look like this:
	```yaml
	<path-to-kubeconfig-file>
	clusters:
	- name: name-of-remote-imagepolicy-service
	  cluster:
	    certificate-authority: /path/to/ca.pem
	    server: https://images.example.com/policy
	users:
	- name: name-of-api-server
	  user:
	    client-certificate: /path/to/cert.pem
	    client-key: /path/to/key.pem
	```

3. Enabling the ImagePolicyWebhook in the kube-apiserver
	if the kube-apiserver runs as a service, you can configure it with the following flags:
	```sh
	ExecStart=/usr/local/bin/kube-apiserver \
	...
	  --enable-admission-plugins=ImagePolicyWebhook \
	  --admission-control-config-file=/etc/kubernetes/admission-config.yaml
	...
	```

	If the kube-apiserver is deployed as a static pod (for example, in a kubeadm-based setup), include similar flags in the pod manifest:
	```yaml
	apiVersion: v1
	kind: Pod
	metadata:
	  creationTimestamp: null
	  name: kube-apiserver
	  namespace: kube-system
	spec:
	  containers:
	  - command:
	    - kube-apiserver
	    - --authorization-mode=Node,RBAC
	    - --advertise-address=172.17.0.107
	    - --allow-privileged=true
	    - --enable-bootstrap-token-auth=true
		
		# config:
	    - --enable-admission-plugins=ImagePolicyWebhook
	    - --admission-control-config-file=/etc/kubernetes/admission-config.yaml
		
	  image: k8s.gcr.io/kube-apiserver-amd64:v1.13.3
	  name: kube-apiserver
	```

# Use Static Analysis of User Workloads
Kubesec scans Kubernetes YAML manifests and gives them a security score with recommendations.
Create `deployment.yaml`: 
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          securityContext:
            privileged: true
```

```sh
kubesec scan deployment.yaml
```
You will receive JSON similar to:
```json
[
  {
    "object": "Deployment/nginx.default",
    "valid": true,
    "score": -30, //Score
    "scoring": {
      "critical": [
        {
          "selector": "containers[].securityContext.privileged == true",
          "reason": "Privileged containers can access the host"
        }
      ],
      "advise": []
    }
  }
]
```
A negative score means dangerous security settings were found.
Improve the Manifest:
```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 10001
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop:
      - ALL
  seccompProfile:
    type: RuntimeDefault
```

# Scan images for known vulnerabilities (Trivy)
Trivy is an open-source security scanner from Aqua Security. It is used to scan:
- Container images
- Filesystems
- Git repositories
- Kubernetes clusters
- IaC files like Terraform, Kubernetes YAML, Helm
- SBOMs
- Secrets
- Vulnerabilities in packages and dependencies

```sh
trivy image nginx:1.18.0
# This scans the Docker image nginx:1.18.0 for vulnerabilities.

trivy image --severity CRITICAL nginx:1.18.0
# Scans only CRITICAL vulnerabilities.

trivy image --severity CRITICAL,HIGH nginx:1.18.0
# Scans only CRITICAL and HIGH vulnerabilities.

trivy image --ignore-unfixed nginx:1.18.0
# Ignores vulnerabilities that do not have a fix available.

docker save nginx:1.18.0 > nginx.tar
# Exports the Docker image to a tar archive file.
trivy image --input nginx.tar
# Scans an image from a local tar archive.

trivy image --severity HIGH,CRITICAL --exit-code 1 nginx:1.18.0
# Fail pipeline if HIGH or CRITICAL vulnerabilities exist:

trivy config deployment.yaml
# Scan Kubernetes YAML files

trivy config ./k8s
# Scan a directory of Kubernetes manifests

trivy config ./terraform
# Scan Terraform code

trivy fs .
# Scan filesystem
 
trivy repo https://github.com/aquasecurity/trivy
# Scan Git repository

trivy k8s --report summary cluster
# Scan Kubernetes cluster

trivy k8s --namespace default all
# Scan a namespace

trivy image --format cyclonedx --output sbom.json nginx:1.18.0
# Generate SBOM

trivy fs --scanners vuln,secret .
# Scan for vulnerabilities and secrets
```
